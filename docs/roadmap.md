# 实施顺序

## 产品里程碑

| # | 名称 | 一句话 |
|---|------|--------|
| 1 | **重构与浏览器扩展** | 按 **DDD** 重组项目，能力边界清晰；主交付改为 **可安装的浏览器扩展**，能力与当前说明一致。 |
| 2 | **选取范围** | 复杂页面上尽量能指到真正要改的区域，减少「看得到选不中」。 |
| 3 | **设置与提示词模板** | 默认复制结构可改、段落可关、可存多套，不必所有人同一套拼法。 |
| 4 | **样式与高级提示词** | 对照观感、在工具里试调样式，导出信息更完整、更利于 AI 改界面。 |
| 5 | **设计系统** | 有团队规范时，可从规范选用或对齐，再带入给 AI 的提示。 |

**远期**：用户手建 **交互流图** → 生成提示词（不自动录制真实操作链路）。

各里程碑的 **要做的事 / 验收目标** 见 [capability-backlog.md](./capability-backlog.md)。

---

## 工程阶段（对内）

| 阶段 | 产出 |
|------|------|
| A | 按领域拆分模块与分层；可注入页面的脚本包；行为与 README 一致 |
| B | 扩展清单、图标、如何装进浏览器、如何唤起选取 |
| C | 自里程碑 2 起的功能按版本迭代 |

**对应关系**：里程碑 **1** 覆盖 **A + B**；里程碑 **2–5** 主要在 **C** 中排期。**3** 可与 **2** 后半并行，但须在「提示词生成」相关边界在代码里划清后再上模板与设置，避免返工。

**依赖**：B 依 A；C 依 B；**4、5** 依赖 **2** 的选取稳定；**3** 不依赖 **4**（模板可先于样式增强上线）。
